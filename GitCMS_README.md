# GitCMS — Git-backed HTML Fragment CMS

GitCMS is a lightweight, single-file admin tool for editing HTML fragments in a GitHub repository.

The repository remains the source of truth. There is no database. Content changes are committed directly to Git.

---

## Current stable baseline

Current stable admin file:

```txt
gitcms_stage2_2_2_sha_session_model.html
```

This version includes:

- single-file OOP architecture
- GitHub repo connection
- repo-level `gitcms.config.json`
- fragment listing
- fragment open/edit/preview/commit
- create/delete fragment
- media drawer
- media upload
- insert image into editor
- SHA/session-based fragment editing

---

## Non-negotiable architecture rule

> The architecturally correct solution is SHA-based Git state, not branch-path refetching.

This is a **must** for GitCMS.

GitCMS must not rely on repeatedly fetching a file by branch path such as:

```txt
contents/src/partials/bob.html?ref=_content
```

as the main way to determine whether the editor has the newest version.

A branch name is a moving pointer. It can produce stale or misleading reads in a browser/API workflow.

Instead, GitCMS must track exact Git state:

```txt
branch → commit → tree → blob
```

For every opened fragment, the admin should know:

```txt
file path
base commit SHA
base blob SHA
current blob SHA
original text
current editor text
last committed text
```

After a successful commit, the editor state should be updated from the commit/blob result, not by blindly refetching the same branch-path URL.

---

## Why SHA-based state is required

A CMS must never show outdated content after a successful commit.

Wrong model:

```txt
open file by branch path
edit
commit
refetch same branch path
hope newest content appears
```

Correct model:

```txt
open file from exact blob SHA
edit
check latest branch head
check whether the same file blob changed remotely
commit on top of latest branch head
update local session state from the confirmed commit result
```

This allows GitCMS to distinguish between:

```txt
safe case:
someone changed config/media/another fragment
you changed bob.html
→ save should continue cleanly

real conflict:
someone changed bob.html
you also changed bob.html
→ show conflict
```

## Recommended repo structure

```txt
repo/
  src/
    index.html
    styles.css
    partials/
      aha.html
      bob.html
    media/
      image.png

  admin/
    gitcms.html

  scripts/
    build-partials.mjs

  gitcms.config.json

  .github/
    workflows/
      pages.yml

  dist/
    generated output only
```

`dist/` should be generated by the build process and should normally be ignored in Git.

---

## `gitcms.config.json`

Recommended config:

```json
{
  "version": 1,
  "content": {
    "partialsDir": "src/partials",
    "partialLinkRel": "partial"
  },
  "media": {
    "mediaDir": "src/media",
    "publicPath": "./media/",
    "maxUploadBytes": 5242880,
    "allowedExtensions": ["jpg", "jpeg", "png", "webp", "gif"]
  },
  "preview": {
    "css": []
  },
  "build": {
    "sourceDir": "src",
    "outputDir": "dist"
  }
}
```

Important distinction:

```txt
media.mediaDir    = where media files are stored in the repo
media.publicPath  = what gets inserted into HTML
```

For GitHub Pages project sites, this is usually correct:

```txt
./media/
```

Avoid this unless the site is deployed at the domain root:

```txt
/media/
```

## Source partial placeholder

During development or source editing, `src/index.html` can use:

```html
<link rel="partial" href="./partials/aha.html" />
```

A build script should replace this placeholder with the actual partial HTML before deployment.

Runtime `fetch()` can be used only for testing. Production should deploy static built HTML.

---

## Build/deploy concept

Recommended flow:

```txt
GitCMS edits src/partials/* and src/media/*
        ↓
GitCMS commits to GitHub
        ↓
GitHub Actions runs build
        ↓
build script creates dist/
        ↓
GitHub Pages deploys dist/
```

This gives:

- static final HTML
- better SEO
- no runtime content-fetch dependency
- visible build logs
- predictable deploys

---

## GitHub Pages workflow

Pages should be set to:

```txt
Settings → Pages → Build and deployment → Source → GitHub Actions
```

The workflow should deploy `dist/`, not `/docs`.

The workflow should trigger on the active content branch, for example:

```yaml
on:
  push:
    branches:
      - main
      - _content
  workflow_dispatch:
```

## Testing checklist

After any major change, test:

```txt
connect
load config
list fragments
open fragment
edit fragment
preview fragment
commit fragment
switch away and back
confirm newest committed content is shown
create fragment
delete fragment
upload media
insert image
copy media path
save config
refresh browser
reconnect
real conflict handling
GitHub Actions deploy
public Pages URL
```

The most important test:

```txt
open partial A
edit partial A
commit partial A
open partial B
return to partial A
partial A must show the newest committed content
```

If that fails, the Git state model is broken.

---

## Product boundary

GitCMS should stay narrow:

```txt
a lightweight Git-backed fragment editor for static/client sites
```

It should not become a full page builder too early.

The strongest product direction is:

```txt
repo as database
HTML fragments as editable content blocks
Git history as audit log
GitHub Actions as build/deploy layer
```
